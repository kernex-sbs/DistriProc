# Phase 03 Plan 01: CRIU Basic Verification

## Goal
Verify that CRIU (Checkpoint/Restore In Userspace) works correctly on this system (Arch Linux, Kernel 6.18) before attempting complex lazy-pages integration.

## Context
Phase 3 aims to use CRIU's `lazy-pages` feature to fetch memory from our remote page server.
Before we can do that, we must ensure basic checkpoint/restore works.
CRIU is already installed (Version 4.2).

## Tasks

- [ ] **1. Create Target Process (`src/test_loop.c`)**
  - A simple C program that:
    - Prints its PID.
    - Loops indefinitely.
    - Prints a counter every second.
    - Modifies a variable in memory (to verify state restoration).

- [ ] **2. Perform Checkpoint**
  - Run `src/test_loop`.
  - Create a directory `images/`.
  - Use `criu dump` to checkpoint the process tree.
  - Verify dump files exist.

- [ ] **3. Perform Restore**
  - Kill the original process (if `criu dump` didn't kill it, though default behavior is to kill).
  - Use `criu restore` to restart the process from the images.
  - Verify the counter continues from where it left off.

## Technical Details

- **CRIU Requirements**:
  - May require root privileges (`sudo`).
  - Requires specific kernel options (usually enabled in standard Arch kernels).
- **Commands**:
  - `criu dump -t <PID> -D images/ -j -v4`
  - `criu restore -D images/ -j -v4`

## Verification Steps
1. Start `test_loop`, observe counter (e.g., 1, 2, 3...).
2. Dump at counter = 5.
3. Restore.
4. Observe output resumes at 6, 7, 8...
